# Datpaq CLI Auth — Website Contract

This document specifies the website-side endpoints the **datpaq CLI** depends
on to perform browser-based OAuth login via Clerk. It lives in the CLI repo
(`github.com/datpaq/cli` → `docs/cli-auth-spec.md`) because the CLI is the
consumer of the contract; the website team implements against it.

## Overview

The CLI launches a one-shot local HTTP listener on a random port, opens the
user's browser to a page on datpaq.com that's gated by Clerk SSO, and waits for
the browser to be redirected back to localhost with a short-lived authorization
code. The CLI then exchanges the code for the user's API key.

```
┌────────┐  1. open browser   ┌──────────────┐
│  CLI   │ ─────────────────▶ │ datpaq.com   │
│ :PORT  │                    │  /cli/auth   │ ── Clerk SSO ──┐
└────────┘                    └──────┬───────┘                │
    ▲                                │ 2. issue code          │
    │ 3. 302 to localhost            │                        │
    │    /callback?code=...&state=...│                        │
    └────────────────────────────────┘                        │
                                                              │
    4. POST /api/v1/cli/auth/exchange {code, state}           │
       ──────────────────────────────────────────────────────▶│
    5. ← {api_key, user: {email, id}}                         │
                                                              ▼
                                                         persist key
                                                       to ~/.config/
                                                       datpaq/config.toml
```

## Endpoint 1 — `GET /cli/auth`

**Purpose:** Browser-facing page that authenticates the user via Clerk, then
hands a one-time code back to the CLI's localhost listener.

**Query parameters:**

| Name    | Type   | Required | Description                                                                                  |
| ------- | ------ | -------- | -------------------------------------------------------------------------------------------- |
| `port`  | int    | yes      | Localhost port the CLI is listening on (e.g. `54321`). Validate range `[1024, 65535]`.       |
| `state` | string | yes      | Opaque nonce generated by the CLI. Treat as an opaque string; max length 128 chars.          |

**Behavior:**

1. If the user is not signed in to Clerk, redirect through the standard Clerk
   sign-in flow (google, github, email/password). Use the current page URL
   (including `port` and `state`) as the post-sign-in redirect target.
2. Once authenticated, generate a one-time **authorization code** (≥128 bits of
   entropy, base64url). Store server-side keyed by `code` with:
   - `user_id` (Clerk user id)
   - `state` (the value from the query string — must match on exchange)
   - `expires_at` (now + 60 seconds)
   - `consumed` (bool, default false)
3. Issue an HTTP 302 redirect to:
   ```
   http://localhost:{port}/callback?code={code}&state={state}
   ```
4. Render a small fallback HTML page with the same callback link, in case the
   browser blocks the redirect or the user's local listener isn't reachable.
   Suggested copy:
   > **You're signed in.** The CLI should pick up automatically. If nothing
   > happens, you can close this window.

**Security notes:**

- The `port` parameter must be on `localhost` / `127.0.0.1` — never trust it for
  an absolute URL. Always render the redirect as `http://localhost:{port}/...`.
- The `state` parameter is opaque to the server; it's the CLI's CSRF defense.
- Reject `port` values outside the unprivileged range to prevent the page from
  being used as a generic redirector.

## Endpoint 2 — `POST /api/v1/cli/auth/exchange`

**Purpose:** Exchange a one-time code for the user's persistent API key. Called
by the CLI from the localhost callback handler.

**Request:** `Content-Type: application/json`

```json
{
  "code": "OBjFf...base64url",
  "state": "n1G2k...opaque-nonce"
}
```

**Response (200):**

```json
{
  "api_key": "dp_live_...",
  "user": {
    "id": "user_2abc...",
    "email": "jerod@datpaq.com"
  },
  "expires_at": null
}
```

- `api_key`: The user's primary API key. If the user has multiple keys, return
  the most recently used one (or create a key named "CLI" on first login).
- `expires_at`: ISO 8601 timestamp, or `null` if the key does not expire.

**Response (400)** — code expired, already consumed, or state mismatch:

```json
{ "error": "invalid_grant", "message": "Authorization code expired or already used." }
```

**Server-side logic:**

1. Look up `code` in the store.
2. Reject if not found, if `consumed` is true, or if `expires_at` is in the
   past.
3. Reject if `state` doesn't match the value stored at issue time.
4. Mark `consumed = true` atomically before returning the key (prevents replay
   if the response is observed in transit).
5. Return the user's API key.

**Auth on this endpoint:** None. The one-time code IS the credential.

## Open questions (please confirm before implementing)

1. **API key shape:** Is there a single primary key per user, or many named
   keys? If many, should the CLI create/use a key named `cli`?
2. **Key visibility on dashboard:** Should keys issued via CLI login show up
   in the user's dashboard with a label like "Created by CLI"?
3. **Rotation:** Do we want a `datpaq auth rotate` command later that hits an
   endpoint to rotate the bound key? Out of scope for v1.
4. **CORS:** Not needed — the localhost callback is a server-to-server-style
   redirect through the browser, not an XHR.

## CLI implementation (already in progress)

CLI side lives in `internal/cli/auth_login.go` (this repo)
(new file). It will:

- Start `http.Server` on a random free port (`net.Listen("tcp", "127.0.0.1:0")`)
- Open `https://datpaq.com/cli/auth?port={port}&state={nonce}` via the OS
  default browser
- Wait up to 5 minutes for the `/callback` request
- Validate state, POST to `/api/v1/cli/auth/exchange`, persist the key via
  `cfg.SaveCredential(api_key)`
- Show "✓ Logged in as {email}" and return

Fallback paths preserved from the generated scaffold:
- `datpaq auth set-token <key>` — paste from dashboard
- `DATPAQ_API_KEY=...` env var — CI / scripted usage
